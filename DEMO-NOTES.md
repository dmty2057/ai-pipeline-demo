# DEMO-NOTES — шпаргалка для виступу

Точний перелік усіх демо-артефактів: гілка, що закладено, що має знайти/прокоментувати Claude,
і чи відкривати PR наживо. Оновлюється після кожного завдання.

> ⚠️ Головне правило: PR для **живого code review** (`demo/code-review`) відкриваю Я САМ НА СЦЕНІ.
> Агент його лише підготував (гілка запушена, PR НЕ відкритий) — інакше рев'ю спрацює завчасно
> і «вау» наживо зникне.

---

## Завдання 2 — непокритий ризиковий шлях (Coverage Summary)

**Файл:** `demo/payments.py` — фінансовий модуль із 3 функцій: `charge`, `refund`, `apply_late_fee`.
**Тести:** `tests/test_payments.py` покриває **лише** успішний шлях `charge()` (відомий користувач, позитивна сума).

**Що НЕ покрито (підтверджено `pytest --cov`, payments.py = 35%):**
- `charge()` — обидві гілки помилок: невідомий користувач (рядок 10) і некоректна сума (рядок 12).
- `refund()` — **повністю без тестів** (рядки 21–26). Це шлях повернення коштів.
- `apply_late_fee()` — **повністю без тестів** (рядки 34–40). Розрахунок пені.

**Чому це ризик (меседж для сцени):** дві з трьох функцій, що рухають гроші, не перевірені жодним
тестом. Баг у поверненні коштів або в нарахуванні пені = прямі фінансові втрати або скарги клієнтів,
і він пройде в продакшен непоміченим. Coverage summary покаже не «52%», а конкретний змістовний ризик.

## Завдання 3 — гілка для живого code review ⭐ ЦЕНТРАЛЬНИЙ МАТЕРІАЛ

**Гілка:** `demo/code-review` (від main). **PR НЕ відкритий** — відкриваю сам на сцені.
**Файл:** `demo/orders.py` (новий), функція `notify_order_shipped(user_id, order_summary)`.

Закладено **рівно дві** проблеми:

1. **Реальний баг — None-dereference (рядок 8).**
   `user = get_user(user_id)` повертає `None` для невідомого `user_id` (тип `Optional[User]`,
   реалізація `_USERS.get(...)`). Наступний рядок `f"...{user.email}..."` звертається до `.email`
   без перевірки на `None` → `AttributeError: 'NoneType' object has no attribute 'email'`
   для будь-якого неіснуючого користувача. Це не надуманий — рівно той самий патерн, що в `charge`/`refund`,
   але там є `if user is None`, а тут його забули.

2. **Порушення конвенції — camelCase (рядок 8).**
   Змінна `shippingMessage` названа в camelCase, тоді як уся кодова база — snake_case
   (`user_id`, `order_summary`, `base_amount`, `days_overdue`). Має бути `shipping_message`.

**Що сказати зі сцени:** «Я відкриваю PR — і за ~1.5–2 хв Claude сам залишає інлайн-коментар
прямо на рядку 8: ловить потенційний None-краш і вказує на порушення стилю іменування».
(Перший прогін довший — підтягується плагін code-review.)
